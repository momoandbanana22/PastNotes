# PastNotes 開発ルール

## 1. 厳密な TDD

**RED → GREEN → REFACTOR の順序を必ず守る。**

1. **RED**: 失敗するテストを先に書く。テストが失敗することを実行して確認してから次へ進む。
2. **GREEN**: テストを通す最小限の実装を書く。テストが通ることを実行して確認してから次へ進む。
3. **REFACTOR**: テストが通ったままコードを改善する。

- バグ修正では「そのバグを再現するテストを先に書く」。テストが RED になることを確認してから修正する。
- リファクタリングでは新規テストは不要。既存テストが GREEN のまま維持されることで安全を確認する。
- テストを書かずに実装を先に書いてはならない。

## 2. 横展開の確認

**バグ修正・リファクタリング後、必ず同じパターンがコードベース全体に存在しないか検索する。**

- 修正を加えたら、同じコードパターンを Grep で検索し、他のファイル・クラスへの横展開が必要かどうか確認する。
- 横展開が必要な箇所が見つかった場合は、同一コミットで全て対処するか、IMPROVEMENT_PLAN.md に記録する。
- 「このファイルだけ直せばよい」と判断する前に、必ず検索を実施する。

## 3. 根本原因の特定

**例外処理・エラーハンドリングの修正では、対症療法でなく根本原因を修正する。**

- 例外処理を修正する前に、例外がどこで発生しどこでキャッチされるか、伝播経路を全て追う。
- 個別箇所への対処で済ませる前に「共通の原因を一か所で直せないか」を必ず検討する。
- 対症療法を選択した場合は、その理由と根本修正が不要な根拠を明記する。

## 4. 事実確認はコマンドで行う

**ファイルの存在・非存在など、コードベースの状態に関する判断は必ずコマンドで確認する。**

- 「このファイルは存在しない」「このメソッドは使われていない」等の判断を、記憶や推測で行ってはならない。
- 判断前に Glob・Grep・Bash 等のツールで実際に確認する。

## 5. 新規ファイル作成前の重複確認

**新規ファイル・クラスを作成する前に、同名のファイル・クラスが既に存在しないかを検索する。**

- ファイル作成前に Glob でファイル名を検索する。
- クラス作成前に Grep でクラス名を検索する。

## 6. チェックリスト項目の確認基準を先に言語化する

**チェックリスト項目を `[x]` にする前に、「何をもって満たしたとするか」の基準を先に言語化し、その証拠を示す。**

- 数値（テスト件数・カバレッジ）は「量」の指標であり、「内容の適切さ」の証拠にはならない。
- 「テスト設計に過不足がないこと」はテスト内容を確認することで判断する。カバレッジで代替しない。
- 「設計に問題がないこと」はコードを実際に読んで確認することで判断する。セッション内の作業の振り返りで代替しない。

## 7. ドキュメントの記載場所

| 内容 | 記載先 |
|---|---|
| 修正すべき問題・改善候補 | `IMPROVEMENT_PLAN.md` |
| 許容した既知の制約・トレードオフ | `IMPROVEMENT_PLAN.md`（DESIGN セクション） |
| リリース前の確認事項（バイナリ判定） | `RELEASE_CHECKLIST.md` |
| 開発手順・環境設定 | `DEVELOPMENT.md` |
| 利用者向け使い方 | `README.md` |

- `RELEASE_CHECKLIST.md` には理由・説明を書かない。確認基準と証拠は `IMPROVEMENT_PLAN.md` に記録する。
- `RELEASE_CHECKLIST.md` は、全ての問題がなくなってリリース可能な状態になったときの**最終確認専用**である。開発中に使用・更新してはならない。
